# Gamma — Architecture & Design Reference

---

## Stack

| Layer | Technology | Purpose |
|---|---|---|
| Game server | Colyseus (Node.js) | Authoritative room state, phase management, matchmaking |
| TV display | Plain HTML + optional Phaser | Central visual output (view screen), shared game state rendering |
| Phone controller | Plain HTML + Colyseus.js client | Per-player input, private game screens |
| Realtime transport | Colyseus WebSocket rooms | Bidirectional state sync and private messaging |
| Client UI / Styling | Svelte + Tailwind CSS (Vite build) | Svelte for UI components and reactivity; Tailwind for utility-first styling. Vite is used as the fast build tool with the Svelte plugin. |

---

## Client Types

Two distinct client types connect to the same Colyseus room and are assigned a role at join time via an `options.role` flag. Both roles are served by a single unified SPA (`client/app/`) that adapts its UI based on the assigned role.

### View Screen (`role: "view_screen"`)
- Runs in a browser on a laptop, TV, or Chromecast
- Renders shared game state for the whole room to see
- Does **not** participate as a player; sends no game input
- Acts as a **secondary authority** when connected — some games require a view screen to function; games that require it must declare `requiresTV: true` in their registry spec
- If the view screen disconnects mid-game the server buffers state; on reconnect the view screen receives a full state snapshot

### Phone Controller (`role: "player"`)
- Minimal UI loaded on each player's phone browser
- Sends input only; never trusts its own game state
- Can receive private messages from the server (hidden instructions, secret roles, private scores)
- Must handle sensor permission prompts (gyroscope, mic, camera) before the round begins

## Landing Page & Session Entry Flow

```
┌─────────────────────────────────────────────────────────────┐
│                     gamma.app  (root URL)                   │
│                                                             │
│   ┌─────────────────────┐   ┌─────────────────────────┐    │
│   │   Open on TV        │   │   Join as Player        │    │
│   │   [Host a Room]     │   │   [Enter Room Code]     │    │
│   └────────┬────────────┘   └────────────┬────────────┘    │
│            │                             │                   │
│     Creates Colyseus room          Joins existing room      │
│     role = "view_screen"           role = "player"           │
└─────────────────────────────────────────────────────────────┘
```

### Host (View Screen) Path
1. Host navigates to `gamma.app` on a browser connected to a TV or shared display
2. Presses **"Host a Room"** — server creates a new Colyseus room and returns a 4-character alphanumeric **room code**
3. View screen enters the **Lobby screen**: room code shown large, player list populates as players join
4. Host selects a game from the game menu (view-screen-only UI)
5. Host can configure game options: round count, time limits, 1v1 bracket vs. free-for-all
6. Host presses **"Start Game"** — room phase changes to `game_loading`

### Player Path
1. Player navigates to `gamma.app/join` on their phone (or scans a QR code shown on the view screen)
2. Enters the **room code** and their **display name**
3. Server validates code → player joins room as `role: "player"`
4. Phone shows the **Player Lobby screen**: waiting list, player avatars, ready indicator
5. Player may tap **"Ready"** — all-ready state is tracked on the server (host can override and force-start)

### Room Code
- 4-character alphanumeric, uppercase, generated by Colyseus at room creation
- Displayed on the view screen at all times during lobby (persistent top-left corner after game starts for late joiners)
- Expires when the room closes

## Room Setup

### Lobby State

```
RoomState {
  phase: "lobby" | "game_loading" | "in_game" | "round_end" | "scoreboard" | "game_over"
  roomCode: string
  hostSessionId: string        // first player (view screen is display-only, never host)
  viewScreenConnected: boolean
  selectedGame: string | null  // registry ID
  gameConfig: GameConfig
  players: MapSchema<PlayerState>
}

GameConfig {
  roundCount: number           // host-configurable, default from game spec
  timeLimitSecs: number        // host-configurable, default from game spec
  matchMode: "ffa" | "1v1_bracket"
}

PlayerState {
  id: string
  name: string
  score: number
  isReady: boolean
  isEliminated: boolean
  bracketSeed: number | null   // assigned at bracket draw
  currentMatchOpponentId: string | null
}
```

### Player Matching for 1v1 Games

When `matchMode: "1v1_bracket"` is set:

1. Server collects all connected players at game start
2. Server performs a **random bracket draw** — shuffles player array with a seeded RNG, pairs by index `[0,1], [2,3], ...`
3. Odd player out receives a bye (auto-advance or waits for round 2 loser's bracket, configurable per game)
4. Each match pair is sent privately to the respective phone screens
5. TV display renders the full bracket and advances it as results come in
6. Winners advance; bracket collapses until one champion remains

```
BracketState {
  rounds: BracketRound[]
  currentRound: number
}

BracketRound {
  matches: Match[]
}

Match {
  id: string
  player1Id: string
  player2Id: string
  winnerId: string | null
  status: "pending" | "in_progress" | "complete"
}
```

### TV / View Screen as Central Authority

Some games explicitly require the view screen to coordinate game state (e.g., Ball Rolling, Sound Replication, Grid Tap Colors). These games set `requiresTV: true` in their registry spec.

- If `requiresTV: true` and no view screen is connected when the host presses Start, the server sends an error and blocks the game from launching
- The view screen may receive and display "hidden" information (e.g., which ball belongs to which player) that phones never see
- View-screen-dependent state is stored in a separate Schema sub-object that only the view-screen client receives

## Game Execution

### Phase Machine

Every game runs through a shared phase pipeline. Individual games extend or skip phases as needed.

```
lobby
  └─► game_loading       (sensors requested, assets preloaded)
        └─► instructions (optional — shown on TV + phone simultaneously)
              └─► countdown
                    └─► in_round
                          └─► round_end
                                ├─► in_round     (next round, if rounds remain)
                                └─► scoreboard
                                      └─► game_over
```

**Phase responsibilities:**

| Phase | View Screen | Phone |
|---|---|---|
| `lobby` | Room code, player list, game picker | Name entry, ready button |
| `game_loading` | Loading animation, game title card | Sensor permission prompts |
| `instructions` | Full-screen instruction animation/video | Mirrored instruction card + "Got it" button |
| `countdown` | 3-2-1 countdown overlay | 3-2-1 countdown overlay |
| `in_round` | Shared game state, scores, timers | Player-specific input UI |
| `round_end` | Round recap, scores delta | Round result card |
| `scoreboard` | Full leaderboard, animations | Rank and score card |
| `game_over` | Final winner reveal, replay prompt | Final rank and play-again button |

### Instructions Phase

Games with non-obvious mechanics use an `instructions` phase before play begins.

- Server transitions to `instructions` phase after `game_loading`
- View screen renders the full instruction screen (animation, diagram, or video clip)
- Each phone receives the same instruction content and displays a mirrored version
- Phone shows a **"Got it / Ready"** button — server waits until all players confirm (or a timeout elapses)
- Some games deliver instructions **staggered** (e.g., Last To Get Instructions) — this is a per-game override of the default broadcast

### Multi-Round Games

Games with `roundCount > 1` cycle through `in_round → round_end` repeatedly.

```
Server round loop:
  for round = 1 to gameConfig.roundCount:
    phase = "in_round"
    await roundComplete signal (all players submitted OR timer expired)
    score all submissions
    phase = "round_end"
    await round_end display timer (3-5s)
  phase = "scoreboard"
```

Round state is preserved across rounds. Player scores accumulate. The server never resets `PlayerState.score` mid-game; it only adds to it.

Round count defaults are defined per-game in the registry spec under `defaultRoundCount`. The host can adjust this in the lobby before starting.

### 1v1 Match Execution

For bracket games, `in_round` is scoped to a single match pair:

1. Server broadcasts active match pairing to all clients
2. View screen shows the match-up and both players' live state
3. Only the two matched players receive active input UI on their phones; others see a spectator view
4. When the match resolves, server records `winnerId` in `Match`, advances bracket
5. If multiple matches run in parallel (e.g., semi-final with 4 players = 2 simultaneous matches), server manages each as a sub-state and waits for all to complete before advancing

## State Management

- **Colyseus Schema** — anything all clients must see: scores, phase, player list, bracket, round timer
- **Server-side Map (private)** — answers before reveal, secret roles, private instructions; pushed to individual clients via `client.send()`
- **TV-only Schema sub-object** — state the view screen needs but phones must not see
- **Client local state** — ephemeral UI state only (animation progress, input buffer); never sent to server

Key rules:
- Server is the only source of truth; clients only react to state deltas
- Never move data into Schema until it should be globally visible
- Host (`hostSessionId`) is just a client with a flag — no special server privileges beyond triggering phase transitions
- Design state to survive disconnects — player data is never deleted on leave; `isConnected` flag is toggled instead
- Rejoin flow: client reconnects with stored `sessionId` → server restores their player slot

## Input Methods

| Input | Reliability | Notes |
|---|---|---|
| Touch / buttons | High | Use everywhere |
| Text input | High | Core party-game mechanic |
| Gyroscope | Medium | iOS needs permission prompt; throttle to ~20 Hz |
| Accelerometer | Medium | Good for shake/threshold; bad for pattern recognition |
| Microphone | Medium | Unreliable in noisy rooms; use amplitude thresholds |
| Camera | Low–Medium | High novelty; needs HTTPS; best for turn-based |
| Vibration | Output only | Android only |

### Motion Detection Limits

- Stick to threshold/binary detection — shake, tilt-held, single-tap
- Avoid rep counting or pattern recognition on raw accelerometer — too noisy, too device-dependent
- For body tracking use TensorFlow.js PoseNet via camera
- Don't send raw gyroscope events — throttle to 20 Hz, dead-zone small movements, calibrate resting position on join

## Key Design Rules

- Don't put data in Schema state until it should be visible to all clients
- Host is a client with a flag — track `hostSessionId` in room state
- Design state to survive disconnects — don't delete player data immediately on leave
- Phaser is only worth adding if there are actual 2D game elements — use plain HTML for pure UI screens
- Games that require a TV must declare `requiresTV: true` and the server must enforce this at start
- Games with 1v1 modes must declare `supportsBracket: true` and implement the bracket interface
- Games with configurable rounds must declare `defaultRoundCount` and `minRounds` / `maxRounds` in their spec
- Sensor permission prompts must be triggered during `game_loading`, not mid-round
